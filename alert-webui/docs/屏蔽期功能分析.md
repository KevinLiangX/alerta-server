在 Alerta 中，屏蔽期 (Blackouts) 的作用是：在指定的某一段时间内，对于符合某些特征（例如特定的环境、服务或资源）的告警，采取静默 / 抑制处理。

这通常用于系统计划内维护 (Scheduled Maintenance) 或者已知问题排查期间，以避免在此期间产生海量、重复的无用告警风暴，干扰运维人员。

一、 怎么操作和使用？
创建屏蔽规则：
在前端左侧导航栏点击 Blackouts，进入屏蔽期页面，点击右下角的 + （加号/添加按钮）创建一个新的规则：

Environment (环境): 比如 Production 或 Development。（必填项，如果不确定可以填大盘里显示的环境名）
Service (服务): 比如 WebAPI 或 Database。（可选）
Resource (资源): 比如某台具体的服务器 IP 或 Pod 名称。（可选）
Event (事件): 比如 CpuHigh。（可选）
Group (分组): 比如 Network。（可选）
Tags (标签): 匹配特定标签的告警。（可选）
Duration (持续时间): 设定这个屏蔽期生效多久，例如 3600 秒 (1小时)。
注意：匹配条件越具体，屏蔽的范围就越小。如果只填了 Environment='Production'，那么整个生产环境的所有新告警都会被屏蔽！

生效过程： 创建完成后，Alerta 后端会自动介入。当监控源（如 Prometheus）发来一个新的告警时，Alerta 在入库前会校验这个告警的属性是否命中了当前仍在有效期内的任何一条 Blackout 规则。

结果体现 (分为两种模式)： 根据 Alerta 后端 

settings.py
 中的配置项 NOTIFICATION_BLACKOUT，会有两种不同的行为（默认为丢弃）：

模式 A（默默丢弃）：如果是默认设置 (NOTIFICATION_BLACKOUT = False)，命中屏蔽期的告警会直接被后端丢弃，并返回 202 Accept 给监控源。意味着你在 WebUI 大盘上根本看不到这条告警。
模式 B（标记为 Blackout 状态但不发通知）：如果配置了 NOTIFICATION_BLACKOUT = True，告警依然会出现在 WebUI 大盘上，但是它的状态 (Status) 会被强制标记为 blackout（这是一个灰色状态）。同时，后端的通知插件（比如发邮件、发 Slack 的插件）会忽略这条告警，实现**“只记录、不打扰”**。
二、 它和 "Shelve (搁置/静默)" 有什么区别？
你可能会疑惑，大盘里对单个告警也有一个带“时钟”图标的操作叫 Shelve（搁置），它俩有什么区别？

Shelve (搁置)：针对已经发生、已经显示在大盘上的具体某一条告警。运维人员在处理的时候觉得：“我知道这个报错了，先让它闭嘴 2 小时，2 小时后如果没修好再亮起来”。这是事后处理。
Blackout (屏蔽期)：针对还未发生、但即将发生的告警规则。通常是提前写好规则：“今晚 2 点到 4 点我要升级数据库，这期间所有从数据库服务发来的告警全部给我拦住”。这是事前预防。、


在 Prometheus + Alertmanager 的生态中，往 Alerta 发送心跳通常有一个非常经典的“套路”，也就是利用 Prometheus 自身发出的 Watchdog（看门狗）告警。

具体原理是：让 Prometheus 故意产生一个永远处于 FIRING（持续报警）状态的测试告警（叫作 Watchdog），然后把它发给 Alertmanager，Alertmanager 再把它路由给 Alerta。 Alerta 的 Prometheus Webhook 接收器（/api/webhooks/prometheus）非常聪明，它内部有硬编码的特殊处理逻辑：只要看到告警名字叫 Watchdog，它就不会把它当成普通的错误告警显示在大盘里，而是自动把它转换成一个 Heartbeat（心跳） 信号更新到后台！

下面是配置这个心跳的完整步骤：

1. 在 Prometheus 端配置 Watchdog 规则
首先，你需要确保 Prometheus 配置里有一条永远触发的告警规则。如果你使用了 kube-prometheus-stack (Prometheus Operator)，这条规则默认就已经存在了。如果没有，你可以手动添加：

yaml
# prometheus-rules.yaml
groups:
- name: general.rules
  rules:
  - alert: Watchdog
    expr: vector(1)
    labels:
      severity: none  # 级别不重要，Alerta会自动处理
    annotations:
      summary: "Prometheus 存活心跳"
      description: "这是一个永远触发的告警，用于确保整个监控流水线 (Prometheus -> Alertmanager -> Alerta) 是畅通的。"
这条规则的 expr: vector(1) 意味着它会永远立刻触发。

2. 在 Alertmanager 端配置路由 (Routing) 并在短间隔内重复发送
因为这不是普通的告警，我们需要它在“持续触发”期间，像心脏跳动一样每隔几分钟就发送一次给 Alerta，否则 Alerta 收不到后续的信号就会认为监控挂了。

yaml
# alertmanager.yml
route:
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 12h  # 对于普通告警，12小时才重发一次，防止骚扰
  receiver: 'alerta-default'
  routes:
  # 将名叫 Watchdog 的告警单独挑出来路由
  - match:
      alertname: Watchdog
    receiver: 'alerta-heartbeat'    # 发给专门用于心跳的 receiver
    group_wait: 0s
    group_interval: 1m
    repeat_interval: 2m  # 关键点！每 2 分钟向 Alerta 发送一次 HTTP POST 刷新心跳
receivers:
- name: 'alerta-default'
  webhook_configs:
  - url: 'http://<ALERTA_IP>:5001/api/webhooks/prometheus'
    send_resolved: true
- name: 'alerta-heartbeat'
  webhook_configs:
  - url: 'http://<ALERTA_IP>:5001/api/webhooks/prometheus'
    send_resolved: false # 心跳本身不可能resolve
3. 配置告警系统的兜底：Alerta 心跳超时怎么报出来？
最后一步，既然监控系统的整个链路（Prometheus ➜ Alertmanager ➜ Alerta）已经打通。如果有一天 Prometheus 进程死锁了，或者网络断了，上面我们配置的“每 2 分钟发送一次”的请求就不来了。 这时候，如我们在上一个回答所说，Alerta 后端依靠 heartbeat 后台定时任务（需要你单独通过 cron 运行 alerta heartbeats，或者直接配置守护进程），一旦发现上次收到来自 Prometheus 的 Watchdog 已经超过预期的时间（比如超过了 5 分钟），插件就会自动在大盘生成一条红色的致命告警，这时候你就会收到通知：“注意！你的监控节点掉线了！”。

这就是整个“监控体系自我检测保活”的最佳实践闭环方案。
